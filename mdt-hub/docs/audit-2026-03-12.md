## 一、问题清单（按严重度）

### 严重（Critical）

- 无。

### 中（Medium）

1. **前端基址推导对 `file://` 打开方式不兼容**
   - 位置：`frontend.html` 初始化脚本
   - 现象：当页面以 `file:///.../frontend.html` 方式打开时，`location.protocol === 'file:'`，原逻辑会拼接错误 API 基址，导致所有请求失败。
   - 影响：本机离线打开前端进行调试/验收时，`/health`、`/agents`、`/discussion/submit` 等请求全链路不可用。

2. **WebSocket 启动提示文案与真实状态机不一致**
   - 位置：`frontend.html` -> `initPage()`
   - 现象：页面显示“WebSocket 已停用”，但实际仍会调用 `connectWS()` 尝试连接。
   - 影响：诊断信息误导，增加排障成本。

### 低（Low）

- 后端 `backend/main.py` 存在 `@app.on_event("startup")` + 模块级 `init_db(); _seed_default_agents()` 双重初始化；当前基于 `create table if not exists` + `insert or ignore` 不会造成功能错误，但有重复执行开销和维护噪音。

### 已核对（无需改动）

- `discussion_submit` 中 `agent_opinion.payload.model_used` 正常写入事件。
- `/agents/{id}/model` 可保存模型并被讨论事件读取。

## 三、接口联通性对照（前端调用 vs 后端行为）

| 接口 | 前端调用方式 | 后端实际行为 | 一致性 |
|---|---|---|---|
| `/health` | 诊断可直接请求（手工/调试） | 返回 `ok/service/db` | ✅ |
| `/healthz` | 诊断可直接请求（手工/调试） | 返回 `ok/service/db/time/version` | ✅ |
| `/models/available` | `GET` 初始化加载模型下拉 | 返回模型列表+默认模型（支持 fallback） | ✅ |
| `/agents` | `GET` 初始化/刷新 Agent 列表 | 返回 `count + agents[]` | ✅ |
| `/agents/{id}/model` | `POST {model}` 保存模型 | 更新并返回 `agent_id/model`（含 enabled 字段） | ✅ |
| `/cases/{id}/documents` | `GET` 加载最近解析结果到三栏面板 | 返回 `documents[].sections` | ✅ |
| `/discussion/submit` | `POST` 提交讨论（带 confirmed_sections/docs_context 开关） | 录入 human 事件 + 生成 agent 事件 + 共识事件 | ✅ |
| `/cases/{id}/events` | 轮询兜底持续拉取 | 返回事件全量列表（按 event_id 升序） | ✅ |
| `/ws/events` | WebSocket 实时订阅 | 连接后通过 broadcast 推送 `mdt_event` | ✅（环境下仅做逻辑验证） |

## 四、端到端流程回归

### 用例 A：导入病历 -> 三栏编辑确认 -> 提交讨论 -> 事件可见

- 通过 `/cases/{id}/documents/upload` 导入文本病历，后端成功抽取 `imaging/labs/medications`。
- 前端三栏可加载并编辑确认项；提交时以 `confirmed_sections` 进入 payload。
- `/discussion/submit` 返回 `accepted=true`，`generated_count` 正常。
- `/cases/{id}/events` 可见 `discussion_message`、多条 `agent_opinion` 与 `consensus_updated`。

### 用例 B：Agent 模型下拉加载/保存 -> `model_used` 入讨论事件

- `/models/available` 加载模型列表成功。
- `/agents/{id}/model` 保存后，`/discussion/submit` 生成的 `agent_opinion.payload.model_used` 与配置一致。

### 用例 C：轮询兜底可持续拉取

- 轮询通过 `/cases/{id}/events` 持续拉取；前端用 `seenEventIds` 去重。
- WebSocket 不可用时，轮询可维持事件可见性。
